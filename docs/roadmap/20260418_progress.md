# ContextHub 进度记录（2026-04-18）

本文件是基于代码仓库现状与 `plan/` + `task-prompt/phase 1-3` 冻结设计的对比总结，作为后续分析的基线。其他 `docs/` 和 `README.md` 内容不作为权威依据（它们基于 plan 写作，存在自由发挥）。

## 先从第一性原理出发：MVP 是要证明什么？

根据 `plan/09-implementation-plan.md` 的 MVP Claim 冻结定义，ContextHub 的核心价值是"在单 OpenClaw 实例 + 多 agent_id 下跑通横向协作闭环"，后置则是"量化收益、显式 ACL 等企业级治理、长期反馈生命周期"。Phase 1-3 的范围是把"MVP 核心 → 企业级治理 → 反馈与长文档"依次补齐；Phase 4 (ECMB) 和 Phase 5 (Production) 才是量化与规模化。

以这个坐标系来看当前代码。

## 一、已完成的 plan 需求

### 1. 存储与信息模型（plan 01 / 02）
- PG 统一存储 + pgvector 向量索引（`alembic/versions/001_initial_schema.py` + `002_force_row_level_security.py`）
- L0/L1/L2 三层内容模型 + `ContentGenerator` / `IndexerService` / `EmbeddingClient`
- 租户 RLS + request-scoped `ScopedRepo` 执行模型（`src/contexthub/db/`）

### 2. 多 Agent 协作闭环（plan 04 / 06）
- 团队层级 + 递归 CTE 可见性（`ACLService`、seed 的 engineering/backend、data/analytics）
- `MemoryService`（私有 → promote → derived_from）、`SkillService`（版本 pinned/latest 解析 + 订阅）
- `PropagationEngine`：`change_events` outbox、periodic sweep、lease 超时 retry、NOTIFY 丢失补扫
- 传播规则：`skill_dep`、`subscription_notify`、`table_schema`、`derived_memory`

### 3. 首个垂直载体（plan 03）
- `MockCatalogConnector` + `CatalogSyncService` + `ReconcilerService`
- `TableSchemaGenerator`、`/api/v1/datalake/*` + `/api/v1/search/sql-context` 结构化组装
- 血缘 / JOIN / 查询模板表打通

### 4. 访问控制与审计（plan 05，Phase 2 范围内）
- `access_policies` + `audit_log` 表（`003_acl_audit_tables.py`）
- 读路径 ACL：`check_read_access`、`filter_visible_with_acl`、deny-override、团队层级约束
- 关键词级 `MaskingService`
- 分层 `AuditService`：`log_strict`（Tier 1 fail-closed）+ `log_best_effort`（Tier 2 SAVEPOINT）
- `ShareService` 纯 ACL share grant（不复制内容）
- Admin API：policies CRUD / audit 查询 / share grant

### 5. 反馈、生命周期与长文档（plan 07 / 11，Phase 3 范围内）
- `context_feedback` + `lifecycle_policies` + `document_sections` 表（`004_feedback_lifecycle_longdoc_tables.py`）
- `FeedbackService`：per-retrieval 幂等（advisory lock）、quality_score、低质量报告、ACL 拦截
- `LifecycleService` + `LifecycleScheduler`：`active ↔ stale → archived → deleted` 状态机、stale 自动恢复、archived 清 `l0_embedding`、跨租户 sweep（对齐 Phase 3 §4.7）
- `PropagationEngine._mark_stale_in_session` 统一委托给 `LifecycleService`（Phase 3 §4.14）
- 长文档管线：`LongDocumentIngester`（PDF / txt / md）、`BaseChatClient` 家族（含 NoOp fail-fast）、`retrieval/long_doc/` 子模块（`coordinator` + `tree_retriever` + `keyword_retriever`）
- `ContextStore.read()` 长文档 L2 从文件系统读取
- Feedback / Lifecycle / Document API、SDK 扩展、`contexthub_feedback` tool 三层接线

### 6. SDK / OpenClaw / 验证（plan 08 / 09 MVP 部分）
- Python SDK + TS Bridge (HTTP sidecar)
- OpenClaw Python Plugin（7 个原始 tool + `contexthub_feedback` 第 8 个）
- Tier 3 功能测试：传播 P-1~P-8、协作 C-1~C-5、可见性 A-1~A-4、反馈 F-1~F-5、生命周期 L-1~L-5、长文档 D-1~D-7 等
- 端到端 demo 脚本（`scripts/demo_e2e.py`、`demo_phase2.py`、`manual_longdoc_smoke.py`）

## 二、未完成的 plan 需求（大颗粒）

## 三、一个第一性的观察

从需求驱动看，现阶段代码已经完整覆盖"MVP Claim + 企业级治理 + 反馈生命周期 + 长文档"这三段落地；真正未完成的是两类：

1. **量化收益的证明**（Phase 4 ECMB）—— 这是"卖点证据"，不是能力本身。
2. **规模化与真实外部系统对接**（Phase 5 + 真实 catalog）—— 这是从 MVP 走向生产的工程化跨度。

这两点都不属于 MVP 退出标准，而是下一阶段的诉求。相对于 plan 的冻结设计，**当前代码已经跑完了 Phase 1-3 的所有范围内需求**，后置项严格对应 `14-adr-backlog-register.md` 或明确写入各 phase plan 的 "3.2 明确不做" 章节，没有漏项。

如果接下来的动作是 Phase 4，建议先回到 `plan/09-implementation-plan.md` 的 "ECMB 设计原则" 把 Tier 1 / Tier 2 的 claim 再确认一遍 —— 尤其是"哪些指标真能独立量化、哪些会和别的变量重叠"（plan 里对实验 4 与实验 2 的重叠已经诚实标注过），避免做一轮没有区分度的 benchmark。
